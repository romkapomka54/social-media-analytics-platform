# IDEAS.md - Ідеї та Нотатки Проекту: Social Media Analytics Platform

Цей документ містить всі зібрані ідеї, детальні нотатки та результати аналізу, які не увійшли безпосередньо до основного плану проекту, але є важливими для його розвитку та подальшого обговорення.

## 1. Загальна ідея проекту
Створити веб-додаток/платформу, яка дозволить користувачам (організаціям) аналізувати коментарі під відео на YouTube, TikTok, Instagram, а також транскрибувати відео з цих платформ. Основна мета — моніторинг громадської думки, аналіз поведінки коментарів (позитивні/негативні) та формування загального уявлення про реакцію аудиторії на контент.

## 2. Ключові функції додатку (Деталізація)

### 2.1. Аналіз коментарів
*   **Збір коментарів:** Реалізувати ефективні механізми для парсингу коментарів, враховуючи API-ліміти та можливі зміни в структурах соціальних мереж.
*   **Аналіз тональності (сентименту):** Використання передових NLP-моделей. Можливо, інтеграція з LLM (NVIDIA GLM-5) для більш точного контекстного аналізу.
*   **Аналіз ключових тем/ключових слів:** Розробка алгоритмів для автоматичного виявлення ключових тем та трендів у великих обсягах текстових даних.
*   **Визначення загальної думки:** Розробка системи для агрегації індивідуальних оцінок тональності та формування узагальненого показника.
*   **Ідентифікація трендів:** Використання часових рядів та ML-моделей для прогнозування та виявлення нових трендів.
*   **Відображення статистики та візуалізацій:** Створення інтерактивних графіків та діаграм для зручного сприйняття інформації.

### 2.2. Транскрибація відео
*   **Розумна економія ресурсів:** Важливо автоматично витягувати існуючі субтитри (якщо вони є) перед запуском транскрибації для економії токенів API та зменшення витрат.
*   **Конвертація мови в текст:** Використання спеціалізованих API для транскрибації.
*   **Ідентифікація різних голосів та форматування діалогів:** На даному етапі це розглядається як додаткова можливість, що потребує більш складних моделей (speaker diarization). Можливе ручне позначення голосів користувачем.
*   **Прив'язка до "тредів повідомлень":** Важливо, щоб транскрибований текст міг бути асоційований з конкретними сегментами відео та їх коментарями.

### 2.4. Автоматизований відділ турботи (Approval Workflow / Human-in-the-Loop)
*   **Виявлення (Polling):** ШІ-агент (OpenClaw) автоматично відстежує нові коментарі через API платформ.
*   **Генерація чернетки (Drafting):** LLM (NVIDIA GLM-5) аналізує контекст (наприклад, скаргу від "детрактора") і формує професійну, персоналізовану відповідь від імені компанії для згладжування конфлікту чи подяки.
*   **Збереження:** Чернетка зберігається у базі даних (Supabase) зі статусом "Очікує підтвердження" без прямої публікації в інтернет.
*   **Схвалення (Approval):** Менеджер компанії заходить у веб-дашборд, бачить список нових коментарів із готовими ШІ-відповідями. Він може відредагувати текст або просто натиснути кнопку "Схвалити".
*   **Авто-публікація:** Після схвалення людиною, агент автоматично відправляє відповідь під відповідне відео/пост у соцмережі.

## 5. Архітектура додатку та Мови Програмування (Додаткові нотатки)

### 5.1. Архітектура: Мікросервіси, ШІ-Агенти та Бессерверні функції
*   **Переваги:** Висока масштабованість, гнучкість, надійність.
*   **Взаємодія:** Мікросервіси взаємодіють через API. OpenClaw взаємодіє з бекендом через Control Gateways для безпечної взаємодії з базою даних та API соцмереж.

### 5.2. Мови Програмування: Python та JavaScript
*   **Вибір:** Python для бекенду/аналітики, JavaScript (з фреймворками React/Angular/Vue.js) для фронтенду, Node.js для API-шлюзів або специфічних бекенд-сервісів.
*   **Причини вибору:** Екосистема Python для Data Science, ефективність JavaScript для веб-інтерфейсів.

### 5.3. Складові додатку (крім автентифікації)
*   **Тенантна архітектура:** Може бути реалізована за допомогою окремих схем баз даних або політики Row Level Security (RLS) у PostgreSQL.

## 6. Безпека, Шифрування та Управління Ризиками ШІ (Додаткові нотатки)
*   **Принцип Zero Trust:** Усі компоненти системи не довіряють один одному за замовчуванням.
*   **Шифрування:** Важливо використовувати сучасні стандарти (AES-256).
*   **Управління секретами:** Зберігання API-ключів у змінних середовища — це мінімальна вимога, можна розглянути використання спеціалізованих сервісів для управління секретами.
*   **Ізоляція OpenClaw:** Деталізація щодо VPS/Docker-контейнерів.
*   **Шлюзи привілеїв:** Важливо, щоб бекенд мікросервіс, який взаємодіє з OpenClaw, мав мінімально необхідні права (принцип найменших привілеїв).
*   **Сертифікація Google:** Розглянути вимоги до безпеки та приватності даних, які Google висуває до додатків, що використовують їхні API.

## 8. Аналіз Конкурентів (Детальніші Нотатки)

*   **Цінова політика:** Дослідити моделі ціноутворення конкурентів (за кількістю згадок, користувачів, обсягом даних, функціоналом) для формування нашої бюджетної та конкурентоспроможної пропозиції.
