# 5. Архітектура додатку (Product Architecture)

Замість використання автономного агента "з коробки", бекенд платформи будується на мікросервісній архітектурі з кастомною оркестрацією на Python. Це гарантує повний контроль над процесами, стабільність роботи та незалежність від конкретного провайдера ШІ.

## 5.1. Мульти-LLM Оркестрація (Шар абстракції LangChain)

Для роботи з генеративним ШІ використовується фреймворк **LangChain** як шар абстракції. Це дозволяє писати код, незалежний від провайдера (провайдер-агностик), і легко перемикатися між моделями (NVIDIA, OpenAI, Anthropic, Google).

Конфігурація ШІ зберігається динамічно у базі даних (Supabase), а не в коді, через систему трьох таблиць:

### `ai_providers`
Зберігає системні облікові дані провайдерів (наприклад, NVIDIA, OpenAI) та їхні зашифровані API-ключі (алгоритм AES-256-GCM). Якщо ключ вичерпано або сервіс недоступний, статус `is_enabled` змінюється на `false`, і система робить автоматичний **failover** (перемикання) на іншого провайдера.

### `ai_models`
Реєстр усіх доступних моделей (наприклад, `z-ai/glm4.7`, `gemini-2.5-flash`, `qwen3.5-122b-a10b`) з їхніми характеристиками (ліміт токенів, підтримка виклику функцій/tools).

### `tenant_ai_configs`
Індивідуальні налаштування для кожної організації (тенанта). Дозволяє вказувати, яку саме модель і який системний промпт (`system_prompt`) використовувати для конкретних задач (use cases), наприклад, для `sentiment_analysis` (дешева модель) або `auto_reply_drafting` (потужна модель).

## 5.2. Фонова обробка та збір даних (Cron & Workers)

Оскільки соціальні мережі (зокрема YouTube) не підтримують вебхуки для коментарів, збір даних відбувається через періодичне опитування (Polling) за допомогою патерну **"True Fire-and-Forget"**.

- **Диспетчер (Dispatcher):** Сервіс, що прокидається за розкладом (наприклад, кожні 10 хвилин), збирає список клієнтських каналів з бази даних і масово розсилає HTTP-запити Робітникам, чекаючи лише підтвердження отримання задачі (`202 Accepted`).
- **Робітник (Worker):** Окремий фоновий процес на Python, який безпосередньо звертається до API YouTube чи Instagram, забирає коментарі та пропускає їх через LangChain-оркестратор для аналізу.
- **Розподілене блокування (Distributed Lock):** Для уникнення дублювання запитів використовується блокування (`withLock`) у базі даних. Це гарантує, що два фонові процеси одночасно не почнуть аналізувати одне й те саме відео.

## 5.3. База даних та Тенантна система (Supabase)

- **Ізоляція даних:** Платформа є B2B SaaS, тому використовується строга мульти-тенантна архітектура. Усі клієнтські таблиці (`users`, `tenant_settings`, `comments`, `videos`) мають обов'язковий зовнішній ключ `tenant_id`.
- **Безпека запитів:** На рівні бази даних і бекенду використовуються функції-обгортки (напр. `tenantQuery`), які не дозволяють розробникам зробити запит без вказівки ID клієнта, що унеможливлює витік даних між компаніями.

## 5.4. Технологічний стек

- **Бекенд та Аналітика:** Python (FastAPI/Flask) + LangChain для збору даних, NLP-аналізу та виклику LLM-функцій.
- **Фронтенд:** JavaScript/TypeScript (React або Vue.js) для побудови веб-дашборду.
- **Сховище:** Supabase (PostgreSQL).
