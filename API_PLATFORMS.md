# API Платформи для Соціальних Мереж

Цей документ містить технічний аналіз API різних соціальних платформ для нашого проекту. Він слугує основою для планування розробки та інтеграції.

## Як читати цей документ

Для кожної платформи ми заповнюємо однакову структуру, щоб мати змогу порівнювати їхні можливості та вимоги.

## 1. Meta (Instagram & Facebook)

### 1.1. Загальна Інформація
*   Офіційна назва API: Instagram Graph API, Facebook Graph API
*   Поточна версія: v25.0 (випущена 18 лютого 2026)
*   Наступні версії: v26.0 очікується пізніше у 2026 році
*   Платформа: Meta for Developers
*   Графік змін (Changelog): [Meta for Developers Changelog](https://developers.facebook.com/docs/graph-api/changelog/)

### 1.2. Автентифікація та Доступ

#### 🔐 Два методи автентифікації

Метод 1: Business Login (Instagram Login)
*   Опис: Пряма автентифікація через Instagram OAuth
*   Коли використовувати: Для додатків, де користувачі входять напряму через Instagram; робота з одним акаунтом
*   Процес:
    1.  Реєстрація додатку в Meta for Developers
    2.  Налаштування Facebook Login продукту
    3.  Отримання Authorization Code через діалог OAuth
    4.  Обмін на короткоживучий токен
    5.  Обмін на довгоживучий токен (60 днів)
*   Необхідні права (scopes):
    *   instagram_basic — базові дані профілю
    *   instagram_graph_user_profile — доступ до інформації користувача
    *   instagram_manage_messages — для роботи з повідомленнями

Метод 2: Facebook Login for Business
*   Опис: Автентифікація через Facebook з підключенням до бізнес-сторінок
*   Коли використовувати: Для платформ з багатьма клієнтами; робота через Business Manager; enterprise-рішення
*   Процес:
    1.  Створення Facebook App в Meta for Developers
    2.  Додавання прав: pages_show_list, business_management, instagram_basic
    3.  Підключення Instagram Business/Creator акаунту до Facebook Page
    4.  Отримання ID Instagram акаунту через Page
    5.  Генерація токенів через обраний метод
*   Вимоги: Instagram акаунт має бути Business або Creator типу та підключений до Facebook Page
*   Термін дії токена: 60 днів (довгоживучий)

Порівняння методів:
| Характеристика | Business Login | Facebook Login |
|----------------|----------------|----------------|
| Складність налаштування | Середня | Висока |
| Найкраще для | Одного акаунту | Багатьох клієнтів |
| Джерело токена | Прямий OAuth через Instagram | Через Facebook Page |
| Права доступу | 3-4 Instagram-специфічні | 3+ Facebook + Instagram |

### 1.3. Ліміти (Rate Limiting)

*   Базовий ліміт: 200 запитів на годину на одного користувача
*   Як це працює:
    *   Кожен Instagram акаунт має ізольований ліміт
    *   Якщо у вас 10 акаунтів, загальна ємність = 200 × 10 = 2000 запитів/год
    *   Ліміт оновлюється щогодини (ковзне вікно)
    *   Всі запити рахуються (включаючи невдалі)
    *   Пагінація рахується як окремі запити
    *   Ліміт прив'язаний до додатку, а не до користувача

*   Формула розрахунку: 200 запитів на годину на одного користувача Instagram

*   Моніторинг лімітів: Використовуйте заголовок X-Business-Use-Case-Usage:
```json
{
"ig_api_usage": [{
"acc_id_util_pct": 50, // % використаного ліміту
"reset_time_duration": 3600 // секунд до скидання
}]
}
```

### 1.4. Майбутні зміни (Важливо!)
*   До 31 березня 2026: Оновити mTLS сертифікати для Webhook.
*   З червня 2026: Старі метрики (Reach, Impressions) зникнуть. Перехід на метрики "Viewers" та "Views". Потрібно буде переробити аналітичні звіти.
*   Січень 2026: Міграція Marketing API на нові об'єкти (`IG User`, `IG Media`) завершена. Ми не спізнилися.

## 2. TikTok

### 2.1. Загальна Інформація
*   Офіційна назва API: TikTok Business APIs (або TikTok Display API для базового доступу)
*   Поточна версія: v2 (актуальна для бізнес-API)
*   Типи API: Існує два основних типи доступу:
    *   *Standard TikTok Connection* (споживчий API): Базова інтеграція, постінг, 4 основні метрики (лайки, коментарі, шери, перегляди).
    *   *TikTok Business Connection* (маркетинговий API): Розширена аналітика, кастомні обкладинки, демографія аудиторії.
*   Графік змін (Changelog): [Посилання на офіційний TikTok Developer Portal]

### 2.2. Автентифікація та Доступ
*   Метод: OAuth 2.0
*   Процес: 
    1.  Реєстрація додатку в [TikTok Developer Portal](https://developers.tiktok.com/)
    2.  Отримання client_key (App ID) та client_secret
    3.  Запит авторизації через https://www.tiktok.com/v2/auth/authorize
    4.  Обмін authorization_code на access_token
*   Термін дії токена: Access token діє 24 години, refresh token може використовуватись для оновлення
*   Необхідні права (scopes):
    *   user.info.basic - базова інформація користувача
    *   video.list - доступ до списку відео
    *   video.upload - завантаження відео (для публікації)
    *   video.insights - розширена аналітика (для Business API)
    *   user.insights - демографічні дані аудиторії

### 2.3. Основні Endpoint'и
*   Інформація про користувача: GET /v2/user/info/ (open_id, union_id, display_name, avatar_url)
*   Список відео: GET /v2/video/list/ (отримати список відео користувача)
*   Деталі відео: GET /v2/video/query/ (детальна інформація про конкретне відео)
*   Коментарі: *(Потребує додаткового дослідження - точний endpoint не знайдено в документації)*
*   Завантаження відео: POST /v2/video/upload/ (публікація відео)

### 2.4. Метрики та Аналітика
| Метрика | Доступ | Опис |
|---------|--------|------|
| like_count | ✅ Standard | Кількість вподобань |
| comment_count | ✅ Standard | Кількість коментарів |
| share_count | ✅ Standard | Кількість поширень |
| view_count | ✅ Standard | Кількість переглядів |
| total_time_watched | ✅ Business | Загальний час перегляду |
| average_time_watched | ✅ Business | Середній час перегляду |
| video_view_retention | ✅ Business | Графік утримання аудиторії |
| Демографія (gender/country) | ✅ Business | Аудиторія за статтю/країною |

### 2.5. Ліміти (Rate Limits)
*   Основний ліміт: Залежить від endpoint'а; для базового доступу ~100-200 запитів на день.
*   Публікація: Строгі обмеження для запобігання спаму.
*   Стратегія: Використовувати експоненційну затримку (exponential backoff) при помилках 429/5xx.

### 2.6. Майбутні зміни та важливі нотатки
*   Webhooks: TikTok підтримує вебхуки для реального часу, але повний каталог доступний після логіну в Partner Center.
*   Коментарі: Документація щодо отримання коментарів обмежена. Можливо, знадобиться використовувати парсинг або моніторинг через Mentions API.
*   Hashtag Listening: Для відстеження хештегів потрібен Business API та налаштування в TikTok Hashtags Manager.
*   Пісочниця (Sandbox): Доступна для тестування через "Generate test access token".

## 3. YouTube

### 3.1. Загальна Інформація
*   Офіційна назва API: YouTube Data API v3
*   Поточна версія: v3 (стабільна)
*   Платформа: Google Cloud Console
*   Графік змін (Changelog): [Google Developers YouTube Changelog](https://developers.google.com/youtube/changelog)

### 3.2. Автентифікація та Доступ
*   Методи:
    *   API Key: Для публічних даних (без авторизації користувача)
    *   OAuth 2.0: Для доступу до приватних даних та дій від імені користувача
*   Процес:
    1.  Створити проект в [Google API Console](https://console.developers.google.com/)
    2.  Увімкнути YouTube Data API v3
    3.  Отримати API Key та/або налаштувати OAuth 2.0 credentials
*   Квота: 10,000 одиниць на день (безкоштовно)
*   Вартість операцій:
    *   Читання (list): 1-50 одиниць
    *   Пошук (search): 100 одиниць
    *   Запис (insert/update/delete): 50-1600 одиниць

### 3.3. Основні Endpoint'и
*   Відео: GET /videos/list (отримати деталі відео, статистику)
*   Пошук: GET /search (пошук відео за ключовими словами)
*   Коментарі:
    *   GET /commentThreads/list (отримати гілки коментарів для відео)
    *   GET /comments/list (отримати всі відповіді на коментар)
    *   POST /commentThreads/insert (додати новий коментар)
    *   POST /comments/insert (відповісти на коментар)
    *   PUT /comments/update (редагувати коментар)
    *   POST /comments/setModerationStatus (змінити статус модерації)
    *   DELETE /comments/delete (видалити коментар)
*   Плейлисти: GET /playlists/list, POST /playlists/insert, POST /playlistItems/insert
*   Канали: GET /channels/list (інформація про канал, статистика підписників)
*   Аналітика: (через YouTube Analytics API)

### 3.4. Параметри запитів
*   `part`: Обов'язковий параметр! Визначає, які групи властивостей повертати (snippet, statistics, contentDetails, status тощо).
*   `fields`: Фільтрує відповідь, дозволяючи отримувати тільки потрібні поля (економить квоту та трафік). Використовує синтаксис, подібний до XPath.
    *   Приклад: `fields=items(id,snippet(title,publishedAt))`
*   `videoId`: Ідентифікатор відео (для коментарів, статистики)
*   `channelId`: Ідентифікатор каналу
*   `maxResults`: Максимальна кількість результатів (1-50)

### 3.5. Ліміти (Quota)
*   Денна квота: 10,000 одиниць за замовчуванням (достатньо для більшості проектів)
*   Моніторинг: Доступний в Google API Console
*   Розширення квоти: Можна подати заявку на збільшення через спеціальну форму

### 3.6. Майбутні зміни та важливі нотатки
*   `comments.markAsSpam`: Цей метод більше не підтримується
*   ETags: Використовуйте для кешування та оптимізації
*   gzip compression: Підтримується для зменшення обсягу даних
*   Webhooks: YouTube не має нативної підтримки вебхуків для коментарів, тому знадобиться періодичне опитування (polling)

## 1.6. Робота з коментарями (Instagram Graph API)
Цей розділ описує критично важливі endpoint'и для реалізації "воркфлоу погодження" (Approval Workflow). Всі запити використовують актуальну версію API v22.0 та нові об'єкти (`IG Media`, `IG Comment`).

#### 🔍 Отримання коментарів
* Метод: GET
* Endpoint: `/{ig-media-id}/comments`
* Опис: Отримує список коментарів для конкретного медіа (поста, Reels).
* Приклад запиту:
  ```
  GET https://graph.facebook.com/v22.0/{ig-media-id}/comments?fields=id,text,timestamp,username,user&access_token={access-token}
  ```
* Параметри:
  * `{ig-media-id}` (обов'язково): Ідентифікатор медіа об'єкта.
  * `fields` (опціонально): Які поля повернути (`id`, `text`, `timestamp`, `user` (новий об'єкт замість `instagram_user`)).
  * `access_token` (обов'язково): Токен доступу.

#### ✍️ Публікація відповіді (КЛЮЧОВИЙ ЕНДПОІНТ)
* Метод: POST
* Endpoint: `/{ig-comment-id}/replies`
* Опис: Публікує відповідь на вказаний коментар. Саме цей запит виконується після того, як менеджер затвердив чернетку, згенеровану ШІ.
* Приклад запиту:
  ```
  POST https://graph.facebook.com/v22.0/{ig-comment-id}/replies?access_token={access-token}&message={URL-кодований_текст_відповіді}
  ```
* Параметри:
  * `{ig-comment-id}` (обов'язково): Ідентифікатор коментаря, на який відповідаємо.
  * `message` (обов'язково): Текст відповіді (має бути URL-кодованим).
  * `access_token` (обов'язково): Токен доступу.

#### 🛡️ Модерування коментарів
* Приховати коментар:
  * Метод: POST
  * Endpoint: `/{ig-comment-id}`
  * Опис: Приховує коментар, встановлюючи поле `hidden` в `true`.
  * Приклад: `POST https://graph.facebook.com/v22.0/{ig-comment-id}?fields=hidden&hidden=true&access_token={access-token}`
* Видалити коментар:
  * Метод: DELETE
  * Endpoint: `/{ig-comment-id}`
  * Приклад: `DELETE https://graph.facebook.com/v22.0/{ig-comment-id}?access_token={access-token}`

#### ⚠️ Важливі зауваження щодо лімітів
* Rate Limits: Кожен з цих запитів споживає ліміт у 200 запитів на годину на одного користувача.
* Оптимізація:
  * Отримання 50 коментарів (1 запит) та відправка 5 відповідей (5 запитів) = 6 запитів.
  * Це дозволяє комунікувати з десятками клієнтів на годину, не виходячи за ліміти.

## 3.7. YouTube Captions API (КЛЮЧОВИЙ ЕНДПОІНТ для економії)
Для реалізації функції "Розумної економії ресурсів" (автоматичне витягування існуючих субтитрів) використовується YouTube Captions API. Це дозволяє отримувати готові текстові транскрипції відео без необхідності транскрибації аудіо.

#### Важливе оновлення (Березень 2024)
> Note: YouTube DEPRECATED параметр `sync` для ендпоінтів `captions.insert` та `captions.update` 13 березня 2024 року. Автоматична синхронізація субтитрів залишається доступною тільки в YouTube Creator Studio.

### 3.8. Методи Captions API
API підтримує наступні методи для роботи з ресурсами `captions`:

#### 🔍 Отримання списку субтитрів
* Метод: GET
* Endpoint: `https://www.googleapis.com/youtube/v3/captions`
* Опис: Повертає список доступних доріжок субтитрів для вказаного відео.
* Обов'язкові параметри:
  * `part`: `snippet` (визначає, які властивості включити)
  * `videoId`: ID відео YouTube
* Авторизація: Потребує OAuth 2.0 зі скоупами `youtube.force-ssl` або `youtubepartner`.
* Приклад запиту:
  ```
  GET https://www.googleapis.com/youtube/v3/captions?part=snippet&videoId={VIDEO_ID}
  Authorization: Bearer {ACCESS_TOKEN}
  ```
* Важливо: Відповідь API НЕ МІСТИТЬ самих субтитрів, тільки метадані про доступні доріжки.

#### ⬇️ Завантаження субтитрів (КЛЮЧОВИЙ ЕНДПОІНТ)
* Метод: GET
* Endpoint: `https://www.googleapis.com/youtube/v3/captions/{captionId}`
* Опис: Завантажує файл субтитрів для конкретної доріжки.
* Параметри:
  * `tfmt` (опціонально): Формат файлу (наприклад, `sbv`, `srt`, `vtt`). За замовчуванням повертається оригінальний формат.
  * `tlang` (опціонально): Мова, на яку потрібно перекласти субтитри (використовується тільки для доріжок `ASR`).
* Авторизація: Потребує OAuth 2.0.
* Приклад запиту:
  ```
  GET https://www.googleapis.com/youtube/v3/captions/{CAPTION_ID}?tfmt=srt
  Authorization: Bearer {ACCESS_TOKEN}
  ```

### 3.9. Важливі обмеження
| Обмеження | Деталі |
|-----------|--------|
| Доступність субтитрів | API працює тільки для відео, які вже мають завантажені субтитри (автоматичні або ручні). |
| Власний канал | Для доступу до субтитрів через OAuth, відео має належати каналу, який авторизував доступ. Для публічних відео можуть знадобитися альтернативні методи. |
| Квоти | Метод `captions.download` має власну вартість квоти (450 одиниць). |
| Формати | Підтримуються формати: `sbv`, `srt`, `vtt`. |

### 3.10. Рекомендація для проекту
1. **Спроба 1 (API)**: Для відео з каналів клієнтів (після авторизації) використовувати `captions.list` + `captions.download`.
2. **Спроба 2 (Публічні відео)**: Для публічних відео без OAuth доступу можна використовувати неофіційні бібліотеки (наприклад, `youtube-transcript-api` в Python) або сторонні сервіси (Apify), але зважати на ризики стабільності.
3. **Fallback (Транскрибація)**: Якщо субтитри відсутні, використовувати транскрибацію аудіо (Whisper) як запасний варіант.

## 5. Архітектурне рішення: Polling vs Webhooks

### 5.1. Загальна стратегія
На основі аналізу API різних платформ, ми приймаємо наступні архітектурні рішення:

| Платформа | Основний механізм | Резервний механізм | Інтервал |
|-----------|-------------------|---------------------|----------|
| Instagram | Webhooks | ❌ Не потрібен | Миттєво |
| TikTok | Webhooks (топ-1000) | API Polling | Webhooks: миттєво<br>API: щоденно |
| YouTube | Polling | ❌ Немає | 5-60 хв (конфігурується) |

### 5.2. Instagram Webhooks
Instagram підтримує вебхуки для отримання сповіщень про нові коментарі в реальному часі.
- **Endpoint**: `POST /webhook/instagram`
- **Payload приклад**:
  ```json
  {
    "object": "instagram",
    "entry": [{
      "time": 1234567890,
      "changes": [{
        "field": "comments",
        "value": {
          "media_id": "123456789",
          "comment_id": "987654321",
          "text": "Дякую за пост!",
          "username": "user123"
        }
      }]
    }]
  }
  ```

### 5.3. TikTok Webhooks + API
TikTok підтримує вебхуки, але з важливим обмеженням: тільки для топ-1000 коментарів за лайками.
- **Endpoint**: `POST /webhook/tiktok`
- **Payload приклад**:
  ```json
  {
    "event_type": "comment",
    "comment": {
      "comment_id": "123456789",
      "video_id": "987654321",
      "text": "Круте відео!",
      "create_time": 1234567890
    }
  }
  ```
- **Резервний механізм**: щоденне API опитування для отримання всіх коментарів.

### 5.4. YouTube Polling
YouTube НЕ ПІДТРИМУЄ вебхуки для коментарів. Єдиний спосіб — періодичне опитування через `commentThreads.list`.
- **Запит**:
  ```
  GET https://www.googleapis.com/youtube/v3/commentThreads
    ?part=snippet,replies
    &videoId={VIDEO_ID}
    &maxResults=100
    &pageToken={NEXT_PAGE_TOKEN}
  ```
- **Оптимізація використання квоти**:
  - Використовувати ETags для кешування відповідей
  - Збільшити `maxResults` до 100 (максимум)
  - Зберігати `nextPageToken` для пагінації
  - Динамічний інтервал залежно від активності відео

### 5.5. Інтеграція з OpenClaw
OpenClaw Gateway має вбудовану підтримку вебхуків через HTTP сервер на порту 18789.
- **Компоненти для реалізації**:
  - Channel Monitor: слухає вхідні webhook події
  - Access Control: перевіряє дозволи перед обробкою
  - Session Manager: spawn isolated session для кожного коментаря
  - Delivery Adapter: форматує та відправляє відповіді
- **Моніторинг**: `openclaw logs --follow journalctl -u openclaw-gateway -f`

## Головний висновок
Ми не можемо мати універсальне рішення для всіх платформ. Кожна платформа вимагає індивідуального підходу, що збільшує складність розробки, але дозволяє оптимально використовувати ресурси та забезпечувати мінімальну затримку там, де це можливо.
